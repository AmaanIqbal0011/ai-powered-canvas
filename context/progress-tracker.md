## Architecture Decisions

- shadcn/ui with Tailwind CSS v4 (@theme inline approach, no tailwind.config.js needed)
- Dark-only theme: all CSS variables are defined in :root with .dark variant class (no light mode)
- Custom theme tokens from ui-context.md mapped to Tailwind utilities via @theme inline
- Generated components/ui/* files are not modified — per ai-workflow-rules.md
- shadcn v4 uses consolidated `radix-ui` package (not individual @radix-ui/* packages) and `data-slot` attributes
- Editor chrome components are "use client" because they manage interactive state (sidebar toggle, tab switching)
- Both components are controlled via props (sidebarOpen/onSidebarToggle, isOpen/onClose) — the page-level layout will wire them together
- Sidebar uses fixed positioning + translate-x for slide animation (does not push page content, per spec)
- Sidebar uses `bg-elevated` for the floating panel surface, navbar uses `bg-surface` for the top bar
- Auth uses Clerk v7 with Next.js 16's `proxy.ts` convention (not `middleware.ts`) — middleware handler protects all routes except `/sign-in` and `/sign-up`
- `ClerkProvider` uses `dark` theme from `@clerk/ui/themes` with CSS variable overrides (no hardcoded colors)
- Auth pages use a simple two-panel layout matching the spec's minimal, professional design
- `UserButton` is placed in the editor navbar right section — Clerk's default menu and profile flows are left intact
- Mock data lives in `lib/mock-data.ts` with a constant `MOCK_CURRENT_USER_ID` for ownership checks
- Dialog state is managed by `useProjectDialogs` hook and exposed via React context so both the sidebar and editor home page can trigger dialogs
- All three dialogs (Create, Rename, Delete) are rendered inline in the context provider to keep dialogs scoped to the provider tree
- Submit calls are stubbed with a 500ms setTimeout — no API calls yet
- Multi-file Prisma schema: generator + datasource in `prisma/schema.prisma`, models in `prisma/models/`
- Prisma v7 PrismaClientOptions requires either `adapter` or `accelerateUrl` — no empty constructor
- `@prisma/adapter-pg` `PrismaPg` factory accepts `{ connectionString }` object and creates the pg.Pool internally
- Client generated to `app/generated/prisma/client.ts` — imported via `@/app/generated/prisma/client`
- Database: Prisma Postgres pooled at `pooled.db.prisma.io:5432` with `postgres://` URL (no `prisma+postgres://` Accelerate)
- Route groups (`(home)` / `(workspace)`) isolate editor home from workspace shell — each has its own layout, avoiding dual-navbar conflicts and letting the workspace page be self-contained with project-aware chrome
- Workspace shell is a client component owning both sidebar states (left project sidebar, right AI sidebar) and wiring project context to the navbar; page content is the pure canvas/AI-slot area
- Access helpers in `lib/project-access.ts` are pure data-check functions callable from any server context — no route handler or component coupling
- Navbar contract: optional props for workspace-only features (projectName, onShare, onAiToggle) keep it usable in both home and workspace modes without branching on route identity
- Sidebar project items are now `<Link>` elements navigating to `/editor/[id]`, making the sidebar the primary navigation between projects
- Collaborators API is a single `route.ts` file handling GET/POST/DELETE under `/api/projects/[projectId]/collaborators` — ownership is enforced server-side for mutations, GET is accessible to both owners and collaborators
- Clerk user enrichment is done server-side in the API route via `batch fetch`: all collaborator emails are fetched in one `getUserList({ emailAddress })` call, then mapped into a lookup — no N+1 loops
- Share dialog is a controlled component: the `WorkspaceShell` manages open/close state and determines `isOwner` from the project list, passing it down to `ShareDialog`
- `useShareDialog` hook is decoupled from the dialog component — it can be reused or tested independently; it manages all API calls, loading states, and the "Copied!" timer
- Avatar component added via shadcn CLI (`npx shadcn@latest add avatar`) — not custom-authored
- Liveblocks Presence includes cursor position (`{ x, y } | null`) and `isThinking` boolean for real-time collaboration state
- Liveblocks UserMeta includes `name`, `avatar`, and `color` — sent as userInfo during ID token authentication
- Cursor color is deterministic: JDK-style hash of the user ID modulo a 10-color palette, ensuring consistency across sessions
- Liveblocks node client is cached on `globalThis` to prevent multiple instances during hot reloads in development
- Auth route uses ID token authentication (recommended over access tokens), requiring manual room creation with `getOrCreateRoom`
- Rooms are created with `defaultAccesses: []` (private) and `usersAccesses` granting `room:write` to the project owner/collaborator — matches the existing project access model
- Auth flow: Clerk identity → project access check → room creation → Clerk user enrichment → ID token with metadata → 403 for unauthorized projects
- `LIVEBLOCKS_PRIVATE_KEY` is read from env (already present in `.env`) and passed as `{ secret }` to `@liveblocks/node`
- Canvas types are defined in a dedicated `types/canvas.ts` module, separate from components and hooks, so both server and client code can reference them without importing React
- The canvas wrapper uses `LiveblocksProvider` with `authEndpoint` pointing to `/api/liveblocks-auth` (the existing auth route) rather than `publicApiKey` — matches the ID token authentication pattern already established
- `CanvasContent` is a child of `ClientSideSuspense`, not passed as a render-prop function, so it renders inline once the Liveblocks room is connected without manual null-checking on nodes/edges
- Error boundary wraps `ClientSideSuspense` to catch Liveblocks connection failures at runtime, while the Suspense boundary handles the Storage-loading state — two distinct failure modes with separate fallbacks
- `useLiveblocksFlow` is used over manual `useStorage` + `useMutation` for nodes/edges because it handles the full React Flow ↔ Liveblocks mapping (onNodesChange, onEdgesChange, onConnect, onDelete) without boilerplate
- Default edge type is `smoothstep` with a subtle arrow marker, matching the edge style defined in ui-context.md — edges are visually secondary to nodes
- Shape rendering uses a `ShapeRenderer` composite component that dispatches to individual shape components by type: CSS-based shapes (rectangle, pill, circle) use styled divs with appropriate border-radius, SVG shapes (diamond, hexagon, cylinder) use inline SVGs with `viewBox="0 0 100 100"` that scale to the node's width/height via `h-full w-full` + explicit container dimensions
- Drag preview ghost is built imperatively (not React-rendered) because the native HTML5 drag API `setDragImage` requires a DOM element — it creates the element offscreen, passes it to `setDragImage`, then cleans it up on the next animation frame
- Text color for node labels is resolved through `getTextColorForFill()` which looks up the fill color in the `NODE_COLORS` palette to find the matching text color, rather than hardcoding text color selection in the node component
- Nodes created via the shape panel drop handler now include explicit `width` and `height` from `SHAPE_DEFAULT_SIZES` so React Flow doesn't need to auto-size after render
- Node resizing uses React Flow's built-in `<NodeResizer>` component which emits "resize" change types — `useLiveblocksFlow` handles these through `applyNodeChanges` which updates the LiveObjects in Storage, so resized dimensions sync across clients automatically
- Label editing uses `useMutation` from `@liveblocks/react` to mutate the `data.label` field on the node LiveObject in Storage — this is preferred over `useReactFlow().updateNodeData()` because it goes directly through Liveblocks Storage (the canonical data source) rather than React Flow's internal store, ensuring proper sync. The textarea's local React state (`editLabel`) is the source of truth during the editing session, and each keystroke also syncs to Liveblocks so collaborators see label changes in real-time
- Placeholder text for empty labels is rendered as an absolute-positioned overlay span below the `ShapeRenderer`, not inside the ShapeRenderer itself — this keeps shape rendering unchanged per scope limits
- Editing interactions are prevented from propagating via `e.stopPropagation()` on `onMouseDown`/`onMouseMove`/`onTouchStart`, which stops React Flow's drag handler from activating while the textarea is focused
- Color toolbar is rendered as a child of the canvas node component (absolute-positioned above it), not as a separate React Flow overlay or external floating element — this scopes it to the selected node's coordinate space and avoids needing to track node positions in a separate layer. Color updates go through `useMutation` to update `data.color` on the node LiveObject, and the text color is derived automatically from `getTextColorForFill()`, so a single mutation toggles both fill and text color in sync
- Custom edge component follows the same architectural pattern as the node component: SVG arrow markers are defined inline in `<defs>` within the edge fragment (two variants — dim `#3a3a42` and bright `#f8fafc`) and referenced via `url(#...)` string on `BaseEdge`'s `markerEnd` prop, since React Flow v12 `BaseEdge` takes a string marker URL, not a `MarkerDefinition` object. Hit area is a transparent 20px-wide `<path>` behind the visible 1.5px stroke, enabling comfortable click/hover targeting without visible thickness increase, matching the spec requirement. Edge labels use the same `useMutation` → LiveObject data update pattern as node labels, with `EdgeLabelRenderer` + `getSmoothStepPath` midpoint coordinates for positioning
- Canvas control bar is rendered as a child of `<ReactFlow>` (inside the children slot) so it can access `useReactFlow()` for zoom handlers — positioned at `bottom-6 left-6` via absolute positioning inside the React Flow viewport, above the shape panel
- Undo/redo hooks (`useUndo`, `useRedo`, `useCanUndo`, `useCanRedo`) come from `@liveblocks/react` (not `@liveblocks/react-flow`) and are called inside `CanvasFlow`, which is already within the `RoomProvider` context — targeting Liveblocks Storage history, not React Flow's internal state
- Keyboard shortcuts hook (`hooks/useKeyboardShortcuts.ts`) is a plain `useEffect` on `window.keydown` with a tag-name check to skip editable fields (`input`, `textarea`, `contentEditable`), decoupled from React Flow's `onKeyDown` prop — ensures shortcuts work regardless of canvas focus state
- Zoom animations use `{ duration: 200 }` passed to React Flow's `zoomIn`/`zoomOut`/`fitView` for smooth feel per spec
